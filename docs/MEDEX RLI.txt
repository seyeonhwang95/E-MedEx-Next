PLEASE NOTE THE FOLLOWING CHANGES:
Addendum No. 4 is issued for the purpose of disseminating Questions and Answers
to questions received from the vendors and other miscellaneous information as
follows:
1 – The Testing Techniques outlined in Exhibit 5 represent the techniques that Broward County
will use in their acceptance testing of each milestone. This has been provided for informational
purposes and does not require a response unless the vendor feels that a particular technique will
not work for testing their proposed solution. Any of these exceptions should be noted in the
vendor’s proposal.
2 - Microsoft Active Directory (AD) is Broward County’s central authentication for access to email,
printers, directories, files and many current applications. Daily account maintenance (upkeep of
user IDs and passwords) with proper procedural checks and balances is needed to assure that
AD is appropriately and timely synchronized with the changing status of the affected users and so
that the risks of social engineering exploits are mitigated. This same effort and attention is
needed for any list of accounts and access rights on the network. Broward County, from the
perspective of both cost and security, requires the “pass-through” authentication using the
accounts and rights already listed in AD for this system.
Broward County has an internal .NET Web Service which will perform “pass-through”
authentication when a username and password is provided. The Web Service response will
contain the pertinent authentication information (i.e., successful or unsuccessful authentication).
Use of this Web Service is the preferred method for “pass-through” authentication and will be
made available to the selected vendor.
3 - Security must be configurable so that a group can be defined and members of that group are
given access to specific screens and functions.
4 – The required solution shall be highly configurable with minimal customization. The main
difference between customizable and configurable is that customization would require the vendor
RFP No. A106025B3
to write custom code for each implementation whereas configurable is something the customer
can modify without much effort.
5- The requirements mention browser-based system. The base solution must be a browserbased
system which follows industry standards as well as Broward County Standards (SOW
Exhibit 3 – OIT STANDARDS).
6- Although the requirements mention solution using either ORACLE OR SQL SERVER database
the vendor should choose one which represents their best solution. The proposal documentation
should be based on that solution; otherwise the vendor would have to submit two proposal
packages (one for each database).
In addition to these items, there are numerous revisions made to this RFP.
Please carefully review this revised RFP in its entirety.

answers to those questions are attached with this Addendum – See

A. Project Overview
Broward County, a political subdivision of the State of Florida, handles and maintains approximately 2000
Medical Examiners, 800 Toxicology, 6000 Cremation and 12,500 Laboratory records per year, plus act as a
Laboratory for Broward’s Sheriff’s Office and Substance Abuse & Health Care Division sample screening. The
estimated total number of internal users is 40 and external users are 30.
The Medical Examiner’s and Trauma Division (METS) of the Human Services Department is charged with the
responsibility for several related functions involving the deaths of citizens, residents, and visitors of Broward
County. This requires that the Division investigates all violent, suspicious, unnatural and unattended deaths;
perform autopsies and other related tests and functions associated with the Office and monitor and track the
results of the specimens and the investigations. The main functions are Investigations; Forensic; Pathology;
Photography; Laboratory; Cremation and Indigent Burials. Currently these functionalities are carried out by an
application written in PowerBuilder 6.5 using SQL Anywhere as a Database engine (implemented in 1999).
The current system partially meets the needs of their business processes. The success of this project will enable
the offices within the METS Division to efficiently meet their obligations in a timely manner with the limited
number of resources available.
According to Strategic Planning Committee recommendations, PowerBuilder platforms, client server
technology will not be pursued as a technical tool by the County. The deployment is a major concern for client
server technology. The County wishes to acquire a web-based case management tool which will improve the
ease of use; provide better integration; implement role based security to protect the sensitive data; and ready
with electronic data exchange. This new system will be a real benefit to Medical Examiner’s and Trauma
Division’s interests and processes, and adaptable for the future needs of the County.
B. Project Goal and Objectives
The goal of this project is to produce the most functionally complete, cost effective, state of the art browser
based Solution. The selected Solution will reduce time spent on manual transactions (by using new technology
such as bar-coding or RFID (Radio Frequency Identifier)) tools. The Solution will enable Medical Examiner’s
office to be integrated with other law enforcement agencies and other county (Medical Examiner’s ) offices.
This new solution will be built on the foundation of the existing system with enhanced security and additional
features as requested by the Medical Examiners & Trauma Division. The details about the tables, fields used in
the existing Database (database diagram) will be provided to the successful Vendor/s.
1. The Solution that may be chosen will provide the following ability to:
a) Track all data modifications, updates, and authoring;
b) Capable of running creative queries using advanced Search Engine;
c) Generate pre-formatted and create ad-hoc reports as needed by management, agents, and other County
personnel;
d) Effectively handle “Police Hold” cases;
3
e) Store photographs outside of the Database using advanced formats such as .JPG, PNG, TIFF and Raw;
f) Use either bar-coding or RFID technology to track samples/evidence;
g) Keep an extensive track of evidence custody chain for all the specimens including deceased’s body;
h) Introduce new processes based upon this electronic system to improve productivity, response time and
electronic workflow;
i) Provide read only access to various law enforcement agencies, counties etc.;
j) Include features like Rich Text format (like Bold, Underline) for comments/notes, e-mails etc.;
k) Capable of using regular, medical and toxicology dictionary for spell check and provide detail data
dictionary;
l) Need to be consistent with the same case numbering system as used in the existing system;
m) No case number should be skipped;
n) Come with the fully automated instrument interface (without manual intervention);
o) Capable of handling e-mails, scanning/storing documents and printing death certificates;
p) Comprised of well written user handbook as well as on-line help;
q) Capable of providing training to technical and non-technical personnel of the County agencies;
r) Must be able to convert prior year data into new system

Testing Techniques
Function
Function testing is performed to ensure that the requirements obtained have been
implemented properly.
Exercise each use-case scenario's individual use-cases flows or functions and features,
using valid and invalid data, to verify that:
o the expected results occur when valid data is used
o the appropriate error or warning messages are displayed when invalid data is used
o each business rule is properly applied
Performance
Performance testing is performed to ensure that response time and time sensitive
requirements have been achieved.
Exercise Function/Business Cycle testing plan under the following conditions and
observe impact:
o normal anticipated workload
o anticipated worst-case workload
Modify data files to increase the number of transactions or the scripts to increase the
number of iterations that occur in each transaction.
Scripts should be run on one machine (best case is to benchmark single user, single
transaction) and should be repeated with multiple clients (virtual or actual).
Key areas of concern that should have been addressed during Requirements
o Event Response time
o Reports run time
o Image display rate
o Batch jobs
Database Testing
Database testing is performed to ensure that the data is being properly managed.
Exercise database access methods and processes independent of the User Interface so you
can observe and log incorrectly functioning target behavior or data corruption.
Invoke each database access method and process, seeding each with valid and invalid
data or requests for data.
Inspect the database to ensure the data has been populated as intended and all database
events have occurred properly, or review the returned data to ensure that the correct data
was retrieved for the correct reasons.

Stress Test/Concurrency:
Stress testing is performed to understand how a system fails under extreme conditions.
Concurrency testing is performed to understand how the system reacts to single objects
being modified by multiple users(deadlocks).
Exercise the appropriate functional test cases under the following stress conditions to
observe failure behavior:
o little or no memory available on the server (RAM and persistent storage space)
o maximum actual or physically capable number of clients connected or simulated
o multiple users performing the same transactions against the same data or accounts
o "overload" transaction volume both on-screen and in interface files
o Reports and printing
Volume Testing
Volume testing is performed to understand how the system reacts to large amounts of
data.
Exercise the appropriate functional test cases under the following high volume conditions
to observe behavior:
o Maximum number of clients connected, or simulated, all performing the same,
worst case (performance) business function for an extended period.
o Maximum database size has been reached and multiple queries or report
transactions are executed simultaneously.
Load Testing
Load Testing is performed to understand how the system reacts under different workloads.
Exercise the appropriate functional test cases under the following conditions:
o Increase in the number of transactions in a test data file.
o Increase in the number of times each transaction occurs.
o increase Workloads
Backup/DR/Recoverability
Recovery tests are performed to ensure that the system can successfully recover from a
variety of hardware and software failures.

Security
Security tests are performed to ensure that application roles have functional access
limited to the appropriate permission level. Security tests also include ensuring that only
appropriate personnel have access to the system.
Exercise the appropriate functional test cases under different user role logins to observe
behavior:
o the expected results occur when the proper role is used
o the appropriate error or warning messages are displayed when an invalid role
attempts execute the test case
o the appropriate error or warning message appears when an invalid user tries to
access the system
Consider the following:
o Test the implementation of our complex password policy (including 45 day rule)
o Sensitive data is encrypted (HIPPA)
o Inquire on Hardware security
o Page Caching
Business Cycle
Business cycle testing is performed to ensure that scenarios performed in their expected
order will produce the appropriate result.
Exercise the appropriate functional test cases under the following conditions:
o Use the normal order of operations (basic flow)
o Use legitimate order of operations that are not as common
o Use legitimate order of operations but include errors (alternate flows)

Configuration Testing
Configuration testing is performed to observe how the software reacts to different
configuration environments (both client and server).
Exercise the appropriate functional test cases on the required hardware and software
configurations under different configurations:
o Server configuration updated with patches or releases
o Client configuration updated with patches or releases
o Client configuration updated with new version of Internet Explorer (for web
based applications only)
o Client configurations with non Internet Explorer browsers (for external web based
applications)
o Client configuration with previous version of Internet Explorer (for web based
applications only)
o Client configuration with dial up access (not on network)
Usability
Usability testing is performed to ensure a pleasant and intuitive user experience.
Exercise the following to observe conformance to Broward County standards:
o Navigation through the application including window-to-window, field-to- field,
and use of access methods (tab keys, mouse movements, quick keys).
o Display controls – review characteristics such as usage, size, position, state, and
focus.
o Colors should be appropriate
o Icons should be intuitive
o Use of Broward County templates provided by Public Communications office (for
applications created by Application Services Division)
Installation
Installation testing is performed to ensure that the install of the system results in a
successful application.
Exercise the installation process under the following conditions:
o New installation of system
o Same or older version already installed
